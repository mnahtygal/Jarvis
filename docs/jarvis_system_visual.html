1. Big Picture: What Jarvis Does

Jarvis takes a user request, decides whether it can answer with a direct skill or memory lookup, gathers useful context from PostgreSQL, sends the question to a local AI model when needed, then saves the conversation for future follow-up questions.

Marty asks → Jarvis brain → Memory + history → Local LLM → Answer saved + returned

3. Main Data Flows

Normal AI Question

  1. User asks a question.
  2. Jarvis saves the user message.
  3. Jarvis detects topic and updates session state.
  4. Context is assembled from memory and history.
  5. llama.cpp answers first; Ollama is fallback.
  6. Response is saved and returned.

Memory Write

  1. User says remember that....
  2. Router detects memory intent.
  3. Memory is inserted or updated.
  4. Audit row is written to memory_history.
  5. Jarvis confirms the memory.

Follow-Up Question

  1. User asks about a topic like Flask.
  2. last_topic is saved as Flask.
  3. User asks “how is it different?”
  4. Jarvis keeps the topic and adds recent history.
  5. LLM answers with context.

4. Built-In Jarvis Commands

Jarvis now includes operational commands that let the assistant explain itself, check itself, report its version, and summarize memory from inside the chat loop.

Command Purpose Module
jarvis help Shows current capabilities and command examples. skills/help_skill.py
jarvis health Runs PostgreSQL, memory, context, llama.cpp, and Ollama checks. skills/health_skill.py
jarvis version Shows version, runtime backend, model, database, and feature status. skills/version_skill.py
jarvis memory summary Summarizes long-term memories, recent history count, and last topic. skills/memory_summary_skill.py
